Authentication should be handled by a dedicated, trusted component or service known as an Identity Provider (IdP). Rather than scattering login logic across every microservice, modern systems centralize this responsibility. You can build your own bespoke IdP, integrate open-source identity frameworks, or rely on fully managed cloud-based solutions (like Auth0 or Okta).

# Authentication

At its core, authentication asks a simple question: *Who are you?* The fundamental challenge in modern web authentication is that HTTP is a stateless protocol. Every network request is essentially a blank slate, carrying no memory of previous interactions. While this statelessness keeps the web fast, scalable, and simple, it means that a user must somehow prove who they are on every single request.

Imagine you want to log in to your bank account. You are presented with a login screen offering a few ways to authenticate:

1. **Using Password Authentication** You enter your username and password. When you hit the login button, your credentials are securely transmitted to the bank’s centralized Auth server.
    
    *(Note: Today, this procedure almost always includes Multi-Factor Authentication (MFA), such as requiring a one-time code sent via SMS or generated by an authenticator app. MFA provides a critical secondary layer of defense, but the underlying architectural flow remains the same).*
    
    The Auth server then checks your username and password against its database to verify that your credentials match. While this sounds simple, doing this at an enterprise scale is incredibly complex. To prevent massive bottlenecks, large tech companies cannot rely on basic database queries. Instead, they utilize a strategic blend of technologies:
    
    - **Bloom Filters:** Probabilistic, memory-efficient data structures that act as a "bouncer," instantly telling the system if a username definitely *does not* exist, saving a trip to the database.
    - **In-Memory Caches:** Systems like Redis handle exact-match lookups in microseconds for recently active users.
    - **Distributed Databases:** Scalable NoSQL systems like Cassandra handle the authoritative checks when caches miss.
    Together, these layers ensure that verifying an identity takes milliseconds, even with billions of users.
    
    See here for more: https://www.youtube.com/watch?v=_l5Q5kKHtR8
    
2. **Using OAuth / Federated Login:** Alternatively, you might choose to log in using a third-party provider (e.g., "Log in with Google" or "Log in with Apple"). This relies on federation protocols like OAuth 2.0 and OpenID Connect (OIDC). Instead of verifying a password, your Auth server redirects the user to the third party, which verifies the user and issues a token back to your system. *(Note: Because building a robust OAuth integration introduces a massive layer of protocol complexity, federated login is out of scope for this specific guide, which focuses on first-party identity management).*

## Basic Authentication

RFC: https://datatracker.ietf.org/doc/html/rfc7617

This section defines the "Basic" Hypertext Transfer Protocol (HTTP) authentication scheme, which transmits credentials as user-id/password pairs, encoded using Base64. This is the simplest HTTP authentication scheme. Here is how it works: you take some user-id and password join them with a colon (e.g. `user-id:password`), encode the result in Base64 and send it in the Authorization header with every request. Here is the critical part: Base64 is not encryption, its **encoding**. It can simply be decoded and viewed by anyone. So why use Base64? Because HTTP headers can only contain certain characters. Base64 converts `user-id:password` into safe characters for transmission; It’s not for security, its for compatibility and anyone can unwrap it. This means if you send basic auth over plain HTTP, you are broadcasting you password in unencrypted form across the network. Basic auth over HTTPs is fine. The TLS encryption protects the credentials, but over HTTP, you might as well post your password on twitter.

There is another issue: you are sending credentials with every single request. That is a lot of opportunities for them to be intercepted or logged somewhere they shouldn’t be. If your credentials are in server logs, cache layers, or proxy logs, that’s a security incident waiting to happen. 

Use Basic Auth for internal tools, local development or simple machine to machine communication where you control the network. For anything else, let’s look at better options.

## **Session-Based Authentication**

While Basic Auth forces the user to send their password on every request, **Sessions** introduce a stateful approach to authentication. In a session-based architecture, the server remembers who is logged in.

Now, every time a user makes a sensitive request to the main server, the client will automatically include the session id in the request which the server (reads the session ID metadata of every request, looks into the database and verifies that the session id is still valid.) will still check against the auth service database to see if it is still valid (calls the auth service to validate the session). Every time a user makes a sensitive request, the client includes the session ID. The server reads this metadata, looks into its session store, and verifies the session is still valid.

Once the server verifies a user's credentials, it creates a record in its own memory (or database) containing the login event and a randomly generated `sessionId`. 

![image.png](https://raw.githubusercontent.com/groove-net/Blogs/refs/heads/main/image.png)

The `sessionId` is just a unique identifier for your login session. The server then sends this unique `sessionId` back to the client. The client stores this ID and attaches it as a "ticket" to every subsequent request. Because the server keeps track of the session, your username and password are no longer required to uniquely identify you after the initial login meaning. The server simply recognizes the user by mapping the session ID to the actual session data (such as user info, roles, or a shopping cart) that lives securely on the server.

The `sessionId` needs to be transmitted back and forth between the client and server. It can either be:

1. **Transmitted via Cookies**
    
    In web browsers, storing the `sessionId` as an HTTP cookie is the industry standard. A browser cookie is a small piece of data (a key-value pair, like `sessionId=abc123`) that a web server asks the browser to store. Once set, the browser automatically attaches this cookie to every subsequent HTTP request sent to that same domain. This makes cookies highly convenient for developers because the transmission lifecycle is handled automatically by the browser.
    
    Storing user identification information as cookies is a browser authentication method known as cookie-based authentication. In this method, the cookie is the storage and transport medium used to store and transport identification which in this case is a session ID
    
    You can see this method in action on most websites that require authentication. In your browser, log in to any application of your choice, open your browser's developer tools, and inspect your network traffic, you will see a newly created secret cookie. The cookie is only available to the application that created it and other websites cannot read it (but they can still attempt to use it; see CSRF below). Because cookies are stored on the client and can potentially be tampered with by users, the server must treat the cookie data solely as a lookup key and never trust it blindly. On the server side, you will need custom middleware to extract the `sessionId` from the incoming cookie and use it to look up the real session state.
    
    However, there are four main ways cookies fall short in modern architectures:
    
    - **Same-Origin rules**: Browsers enforce strict Same-Origin policies to prevent malicious cross-site interactions. However, modern applications often decouple the frontend and backend. For instance, an SPA hosted at `app.example.com` might need to authenticate against an API at `api.example.com`. Because these are different origins, the browser will strictly block cross-origin cookies by default. Allowing this communication requires configuring complex Cross-Origin Resource Sharing (CORS) permissions on the server and ensuring the cookies are explicitly permitted for cross-domain transmission. A large or complex systems may have many servers registered as different domains. This can makes setting up CORS permissions very painful.
    - **Cookies bring in CSRF vulnerabilities:** CSRF fundamentally exploits the "ambient" nature of cookies—the fact that browsers send them automatically, regardless of who initiated the request. In a Cross-Site Request Forgery (CSRF) attack, a malicious website tricks a user’s browser into sending unintended requests to a site where the user is already logged in. A malicious website can embed hidden requests—like a form submission or an image tag—that cause the browser to contact the victim site. Because browsers automatically attaches the user’s cookies, including their session cookies, with every request to your domain, your server assumes the action (like transferring money or changing an email) was authorized. Your server cannot easily distinguish between a legitimate action initiated by the user and a malicious attack initiated by another site. The attacker never sees the cookies but gains the ability to force actions on behalf of the user.
        
        To secure the perimeter against CSRF, you must implement strict `SameSite` cookie attributes or rely on traditional anti-forgery tokens.
        
    - **Cookies bring in XSS vulnerabilities**: Cookies are also heavily targeted during Cross-Site Scripting (XSS) attacks. XSS occurs when an attacker successfully injects malicious JavaScript into a legitimate web page viewed by your users. If a session cookie is stored plainly in the browser, that injected script can access it via `document.cookie` and quietly transmit the session ID to an attacker's server, resulting in complete session hijacking.
        
        To neutralize this threat, your authentication server must instruct the browser to drop the session ID into a cookie with the `HttpOnly` attribute applied. The `HttpOnly` attribute strictly forbids client-side JavaScript from reading the cookie, keeping the token invisible and safe from malicious scripts.
        
    - **Non-browser clients lack native support:** Finally, cookies are primarily a browser mechanism. Non-browser clients, such as mobile apps, server-to-server calls, and third-party integrations, do not possess built-in "cookie jars" to manage state automatically. To use cookie-based authentication, developers of these clients must manually implement complex logic to parse incoming Set-Cookie headers, persist the data, handle expiration times, and manually attach the cookies to all subsequent requests, all while building their own custom protections against CSRF. Because of this heavy operational burden, most third-party APIs assume clients cannot handle cookies safely.
    
    **Security Note:** If you must use cookies for authentication, a production-grade authentication cookie must be hardened using a combination of three security attributes: **`HttpOnly`** to prevent JavaScript access and mitigate XSS, **`SameSite`** to control cross-domain routing and prevent CSRF attacks, and **`Secure`** to ensure the cookie is only ever transmitted over encrypted HTTPS connections.
    
2. **Transmitted via HTTP response objects and headers**
    
    Because cookies are inherently a browser-centric feature, relying on them immediately complicates authentication for mobile applications, command-line interfaces, and server-to-server communication where native cookie management is unsupported. To support these environments, non-browser clients receive the `sessionId`  directly within the JSON body of the HTTP response object.
    
    The non-browser client is then responsible for parsing this response, storing the session ID in its own memory or local file system, and manually attaching it to all subsequent requests. This is typically achieved using HTTP headers. Developers commonly leverage the standard Authorization header with a custom scheme, formatting it as `Authorization: Session abc123`, or they utilize a completely custom header such as `X-Session-ID: abc123`.
    
    This explicit transmission method is highly attractive because it does not rely on client-specific browser mechanisms. Browser clients, single-page applications, and native mobile apps can all consume the exact same authentication API without worrying about CORS cookie restrictions. On the server side, handling this simply requires a custom middleware in your request pipeline that intercepts the incoming HTTP headers, extracts the session ID, and resolves the user's identity before routing the request to your protected endpoints.
    
    However, choosing this method for **browser-based clients** specifically introduces a critical security trade-off:
    
    - **Local storage bring XSS vulnerabilities**: Because the browser will not automatically attach headers to outgoing requests, your client-side JavaScript must store the session ID somewhere it can read it, typically in `localStorage` or `sessionStorage`. This is a fundamentally public space within the browser's execution environment. If your web application suffers a Cross-Site Scripting (XSS) vulnerability, an attacker's injected script can instantly query `localStorage`, steal the session ID, and achieve complete session hijacking. Unlike cookies, which can be shielded from JavaScript using the `HttpOnly` attribute, there is no browser-level defense mechanism to protect data stored in local storage from a successful XSS attack.
        
        If you are building a browser-based application and choose to store tokens in `localStorage` to attach them as headers, how do you defend against XSS? Well, your defense must focus entirely on preventing malicious scripts from ever executing on your page in the first place. You cannot protect the token once the script is running, so you must stop the injection entirely.
        
        The most powerful tool for mitigating XSS is a strict **Content Security Policy (CSP)**. A CSP is an HTTP response header that acts as a ruthless whitelist, explicitly instructing the browser which external domains are allowed to load and execute scripts. By default, browsers trust any script they encounter. With a properly configured CSP, even if an attacker successfully injects a malicious script tag into your page's HTML, the browser will look at the policy, see that the script's source is not whitelisted, and outright refuse to execute it.
        
        Beyond network policies, you must neutralize the injection payloads at the application level through strict **Input Sanitization and Output Encoding**. This means aggressively sanitizing any user-generated content before it touches your database, and strictly encoding that data before rendering it back onto the screen. Fortunately, modern frontend frameworks like React, Angular, and Vue automatically encode variables bound to the UI, heavily mitigating traditional DOM-based XSS out of the box. However, any use of dangerous APIs—like forcefully injecting raw HTML strings into a component—bypasses these native protections and instantly reopens the door to token theft.
        
        Finally, if the permanence of `localStorage` remains too risky for your security posture, the modern industry standard is to store the access token strictly in **JavaScript memory**, such as a private closure or application state. While a highly sophisticated XSS attack can still potentially read memory, it is significantly harder to orchestrate than simply scraping `localStorage`. The architectural trade-off here is that application memory is wiped completely whenever the user refreshes the tab. To prevent forcing the user to log in on every refresh, this in-memory approach is almost always paired with a silent background mechanism—typically a dedicated `/refresh` endpoint secured by an `HttpOnly` cookie—to seamlessly mint a fresh token when the application reloads.
        
        Again, Cross-Site Scripting (XSS) is almost exclusively a browser-centric vulnerability. Native mobile applications, desktop software, command-line tools, and server-to-server microservices do not evaluate untrusted HTML or execute web scripts in a shared environment like a browser does. Therefore, if your client is a native mobile app storing a session token securely in an iOS Keychain or Android Keystore, XSS is simply not a threat vector. The risk of XSS only applies when you are building a web application—such as a Single Page Application—that runs inside a user's web browser.
        

In essence, if you're building stateful session-based auth, you can transmit and persist the `sessionId` however you want, as long as the client attach the session ID with every request and the server can extract that ID and use it to look up the session state in its session store (memory, Redis, DB, etc.)

The server keeps the session active as long as you are interacting with it (i.e. as long as you are making requests). If you exceed a specific period of inactivity (i.e. it has exceeded some time since your last request), the server will notice the time gap (by comparing the `datetime` of your current request with the `datetime` of your last request stored in the user’s session data) and invalidate the session automatically, prompting you to log in again. Critical applications (like banking) use short-lived sessions (~5 minutes), while social media apps use long-lived sessions (months). If the session expires on the server, the ID held by the client instantly becomes worthless.

When a user logs out, the server destroys the session record in its database and optionally instructs the client to delete the locally stored session ID.

### **The Horizontal Scaling problem**

The biggest architectural challenge with stateful sessions is that the actual session data (user info, auth state, cart contents, etc.) lives on the server.

If you have a single server, this is fine. But when your application grows and you place multiple instances of an API server behind a load balancer, you encounter a major routing problem: **Server B does not know about the session created on Server A.**

There are two common ways to handle this:

#### **Option 1: Sticky Sessions**

In this setup, the load balancer is configured to always send the same user to the same server.

- **The Flow:** User connects → Server A creates the session → Load balancer ensures all future requests from that user go only to Server A.
- **The Problem:** This is notoriously difficult to scale. Server A might become completely overloaded while Server B sits idle. Furthermore, if Server A crashes, every single user routed to it loses their session and is instantly logged out; additionally this also means that redeployment and updates without dropping sessions becomes tricky.

#### **Option 2: Distributed Session Storage**

Instead of storing session data in the individual server's memory, all servers read and write to a centralized, high-speed storage location (typically Redis, Memcached, or a distributed SQL database).

- **The Flow:** Server A creates a session and saves it in Redis. The user later hits Server B. Server B takes the cookie, looks up the session ID in Redis, and successfully authenticates the user. This way all servers can read/write to the same session state.
- **The Problem:** While this solves the scaling and crash-loss issues of sticky sessions, it introduces operational complexity. You now have to maintain, secure, and pay for a separate Redis cluster, and every HTTP request now suffers a slight latency penalty as the server makes a network call to the cache.

## Token-Based Authentication **and Stateless Identity Storage**

Traditional session-based authentication requires services to perform a request to the Auth Component on every incoming request to retrieve user claims, roles, and session data. While secure, it introduces server-side overhead because each request depends on an in-memory session-store query or worse an external datastore query and in either you might incur an additional HTTP call if the Auth Component is decentralized; as the user base grows, memory usage and constant queries can become increasingly inefficient.

Token-based authentication addresses this by removing the need for server-side identity storage. Instead of storing user session data with the Auth Component, the system serializes all necessary identity information—claims, roles, permissions, session metadata such as ip address or user agent, and expiration timestamps—into a self-contained token. 

Four Main Cyber Security Services
• Authentication: Guarantee the origin of data
• Integrity: Ensure the data remains intact
• Confidentiality: Permit the secrecy of the data
• Availability: Ensure the data remains accessible

In the world of identity, there is a fundamental choice: do you just want to make sure no one **tampered** with the data, or do you want to make sure no one can **read** it either?

This is the core difference between **Signing** (Integrity) and **Encryption + Signing** (Privacy + Integrity).

| **Feature** | **Signing Only** | **Encryption + Signing**  |
| --- | --- | --- |
| **Philosophy** | "Show your cards, but prove they're real." | "Hide the cards entirely." |
| **Visibility** | **Transparent:** Anyone can Base64 decode it and read your data. | **Opaque:** The payload is gibberish to everyone but the server. |
| **Protection** | **Integrity:** Detects if a user changed their `user_id` or `role`. | **Integrity + Privacy:** Prevents both tampering and information leakage. |
| **Standard** | JSON Web Signature (JWS) | JSON Web Encryption (JWE) |
1. Signing (Integrity)
    
    Standard JWTs (JWS) are like a **Certified Check**. Anyone who holds the check can read the amount and the recipient’s name. However, because of the cryptographic signature at the bottom, they cannot change "$100" to "$1,000" without breaking the "seal."
    
    - **The Risk:** Information Leakage. If you put internal metadata, email addresses, or database IDs in a signed-only token, you are broadcasting that data to the public internet.
    - **The Use Case:** Public APIs or mobile apps where transparency is okay as long as the data is trustworthy.
2. Encryption + Signing (Privacy + Integrity)
    
    The JWE standard is like a **Bank Vault**. You cannot see what is inside the vault, and you certainly cannot change its contents. You need the specific cryptographic key to even begin reading the data.
    
    - **The Benefit:** It prevents attackers from gaining insights into your internal naming conventions or user permissions. Even if they have the token, they have zero context.
    - **The Use Case:** Web applications requiring high security or systems handling sensitive PII (Personally Identifiable Information) that should never hit the browser's memory in plain text.

Summary: Which is right for you?

- **Use Signing (JWS)** when you need a universal, language-agnostic standard for public APIs or mobile clients and don't mind if the claims are readable.
- **Use Encryption + Signing (JWE)** when you want maximum "Defense in Depth" for a web-based ecosystem, ensuring your internal data remains both untamperable and invisible.

### **Signed Tokens**

Once generated, the token is digitally signed, before it’s sent to the client making it easy to verify that the content has not been modified or tampered with by the time it is sent back along with a request. Because the token itself proves the user's identity, authentication becomes fully stateless. The server scales naturally without the need for sticky sessions or distributed caching.

#### JSON Web Tokens (JWT)

The industry standard for token-based authentication is the JSON Web Token ( https://datatracker.ietf.org/doc/html/rfc7519). JWTs are an open standard  that defines a compact and self-contained way for representing and securely transmitting claims and information securely between two parties as a JSON object.

Because JSON parsers are common in most programming languages, you can easily map JSON data directly to objects in almost all programming languages. Conversely, older XML-based standards like SAML doesn't have a natural document-to-object mapping. This makes it easier to work with JWT than SAML assertions on multiple platforms. Additionally, JSON is less verbose than XML, when it is encoded its size is also smaller, making JWT more compact than SAML. This makes JWT a good choice to be passed in HTML and HTTP environments.

While tokens can be fully encrypted to hide their contents, this chapter focuses on **Signed Tokens** (specifically, JSON Web Signatures or JWS). A signed token guarantees the *integrity* of the data contained within it but not its privacy. The server signs the token using a secret key (HMAC) or a public/private key pair (RSA or ECDSA). This signature certifies that the token was issued by your authentication service and has not been tampered with. The token can be verified and trusted because it is digitally signed.

Here are some scenarios where JSON Web Tokens are useful:

- **Authentication and Authorization**: This is the most common scenario for using JWT. Once the user is logged in, each subsequent request will include the JWT, allowing the user to access routes, services, and resources that are permitted with that token, or rather with the claims and roles encoded in that token.. Single Sign On — an authentication approach that allows a user to log in once and then access multiple independent applications or services without needing to re-enter their credentials for each one — is a feature that widely uses JWT nowadays, because of its small overhead and its ability to be easily used across different domains.
- **Information Exchange**: JSON Web Tokens are a good way of securely transmitting information between parties. Because JWTs can be signed—for example, using public/private key pairs—you can be sure the senders are who they say they are. Additionally, as the signature is calculated using the header and the payload, you can also verify that the content hasn't been tampered with.

In its encoded form, a JWT generate by JWS consists of three Base64Url-encoded strings separated by dots (`header.payload.signature`):

1. **Header:** The header typically declares some metadata about the token. Typically the type of the token, which is `JWT`, and the signing algorithm being used by the issuer to generate the token, such as HMAC SHA256 (`HS256`) or RSA (`RS256`).
    
    For example:
    
    ```json
    {
      "alg": "HS256",
      "typ": "JWT"
    }
    ```
    
    Then, this JSON is **Base64Url** encoded to form the first part of the JWT.
    
2. **Payload:** The second part of the token is the payload, which contains the claims. Claims are statements about an entity (typically, the user) and additional data stored as a name-value pair in the JSON format. There are three types of claims: registered, public, and private claims.
    - Registered claims: These are a set of predefined claims which are not mandatory but recommended, to provide a set of useful, interoperable claims. Some of them are: `iss` (issuer), `exp` (expiration time), `sub` (subject), **aud** (audience), and others. The token can be refreshed after expiration to keep it secure during prolonged sessions.
    - Public claims: These can be defined at will by those using JWTs. But to avoid collisions they should be defined in the IANA JSON Web Token Registry or be defined as a URI that contains a collision resistant namespace.
    - Private claims: These are the custom claims created to share information between parties that agree on using them and are neither registered or public claims.
    
    An example payload could be:
    
    ```json
    {
      "sub": "1234567890",
      "name": "Alice",
      "role": "admin",
      "exp": 1733434800
    }
    ```
    
    The payload is then **Base64Url** encoded to form the second part of the JSON Web Token.
    
3. **Signature:** The signature is used to verify the message wasn't changed along the way, and, in the case of tokens signed with a private key, it can also verify that the sender of the JWT is who it says it is.
    
    To create the signature part you have to take the encoded header, the encoded payload, a secret, the algorithm specified in the header, and sign that. Here is how each algorithm works:
    
    - For internal systems where you control both the authentication server and the consuming APIs, the most common algorithm is HMAC SHA256.
        - **HMAC SHA256 (Symmetric Signing)**
            
            HMAC (Hash-Based Message Authentication Code) is a symmetric signing technique that allows two parties who share the same secret key to verify both the integrity and authenticity of a message. This means that the Auth component (which issues the token) and the API servers (which validate the token) must all share the exact same secret key. This secret is typically generated using a Cryptographically Secure Pseudorandom Number Generator (CSPRNG) and securely distributed across your infrastructure. 
            
            With HMAC, the issuer (our Auth component) generates a signature using a combination of the message content and a shared secret. The receivers (our API servers) then recomputes the signature using the same secret. If the signatures match, the message is guaranteed to be unmodified and to have been created by someone who knows the secret key.
            
            At a high level, HMAC generates a signature by feeding the shared secret and the message through a hashing algorithm. 
            
            A simplified conceptual view:
            
            ```c
            signature = HMAC(secret, message)
            ```
            
            Internally, HMAC mixes the secret into the hashing process using two distinct padding steps (`opad` and `ipad`). The internal operation looks like this:
            
            ```c
            HMAC(secret, message) =
                H(XOR(secret, opad) + H(XOR(secret, ipad) + message))
            ```
            
            > `🗒️NOTE`
            The H stands for any cryptographic hash function (such as SHA-256 or SHA-512). So HMAC-SHA256 means an HMAC built using the SHA-256 hash function. HMAC internally uses two constant padding values known as `opad` (outer padding) and `ipad` (inner padding). They are special padding constants used internally to safely mix the secret key with the hash function. Both are fixed, standardized bit patterns defined by the HMAC specification (RFC 2104). They are not secret.
            > 
            
            Unlike a simple hash (e.g., standard SHA256), HMAC cannot be forged by an attacker because the secret key is an essential, unknown variable in the computation. This provides two critical security guarantees: **integrity** and **authenticity**.
            
            Integrity is guaranteed because any modification to the payload claims—such as an attacker changing `"role": "user"` to `"role": "admin"`—will result in a completely different hash output. This means that during verification, your API server will compute a different expected signature from the tampered payload, detect the mismatch, and reject the request. An attacker would need to know the secret key to generate a valid signature for their tampered message.
            
            Consequently, the secret key must never be exposed or leaked. As long as the secret remains secure, authenticity is also guaranteed. Because only your internal systems (the Auth service and the API servers) possess the shared secret, any successfully validated token is definitively proven to have originated from your Auth service as the only token issuer.
            
            In summary, HMAC provides strong integrity and authenticity guarantees using a single shared secret, preventing attackers from tampering with or forging tokens. It is crucial to remember, however, that HMAC provides zero confidentiality. The payload remains fully readable in transit and at rest; only its integrity is cryptographically protected.
            
            **How does this work in the real world?**
            
            A client (browser, mobile app, CLI) sends credentials to your Auth service:
            
            ```bash
            POST /login
            {
              "username": "alice",
              "password": "mypassword"
            }
            ```
            
            Your Auth service verifies the credentials and creates the header:
            
            ```json
            {
              "alg": "HS256",
              "typ": "JWT"
            }
            ```
            
            and a payload (claims), e.g.:
            
            ```json
            {
              "sub": "123",
              "name": "Alice",
              "role": "admin",
              "exp": 1733434800
            }
            ```
            
            Your Auth component has a shared secret key randomly generated using a Cryptographically Secure Pseudorandom Number Generator (CSPRNG) which is shared across all your servers servicing requests at the edge layer:
            
            ```bash
            secret = "my-very-strong-secret"
            ```
            
            It uses it to create the signature by signing the Base64Url encoded header and payload (concatenated using the `.` delimiter) via HMAC:
            
            ```c
            encodedHeader = Base64UrlEncode(header);
            encodedPayload = Base64UrlEncode(payload);
            signature = HMAC(secret, encodedHeader + "." + encodedPayload)
            ```
            
            Then, it Base64Url encodes the signature and appends it to the encoded header and payload using the `.` delimiter to form the final token. Every piece of a JWT is Base64Url-encoded so it can be safely transmitted in HTTP headers:
            
            ```c
            encodedSignature = Base64Encode(signature);
            token = encodedHeader + "." + encodedPayload + "." + encodedSignature;
            ```
            
            This signed token is returned to the client, where the client stores the token depending on the client type—usually in memory or cookie for Browser SPAs, secure storage (like a Keychain) for mobile apps, or application memory/caching for server-to-server communication. Then, every subsequent API request includes the token.
            
            When your API receives a request, extract the token and split it into `encodedHeader`, `encodedPayload` and `encodedSignature` using the delimiter:
            
            ```c
            encodedHeader . encodedPayload . encodedSignature
            ```
            
            The server recomputes the signature using the *same* secret key the Auth service used (hence the *shared* in shared secret key):
            
            ```c
            expectedSignature = HMAC(secret, encodedHeader + "." + encodedPayload)
            ```
            
            Therefore if:
            
            ```c
            Base64UrlEncode(expectedSignature) == encodedSignature
            ```
            
            …then the token is authentic and the payload has not been tampered with. (*Note: Production systems use constant-time string comparisons here to prevent attackers from guessing the signature via timing attacks).* Otherwise, if the signatures do not match, reject the request with `401 Unauthorized`. 
            
            Of course, tokens also expire. So, even though the token happened to be authentic, we must also reject the request when expiration occurs, i.e. if the current date and time have exceeded the Unix timestamp stored in the `exp` claim. In this case, we also reject with `401 Unauthorized`. We provide a reason code in the response body of these rejections to distinguish these errors from one another, e.g. `"token_expired”` or `"invalid_token”`.
            
            In the event the token is completely valid and unexpired, but the user simply lacks the required roles or permissions for that specific endpoint, we return a `403 Forbidden`. Otherwise, we process the request as normal.
            
            Because tokens are completely stateless, managing expiration introduces a strict architectural constraint. Your API server has no active "session" in memory to extend. It cannot magically update the expiration time of an existing token mid-flight. When a token naturally expires, the client must explicitly request a new token from the Auth service—typically via a dedicated refresh token flow—before retrying the original request.
            
            Refresh token flow? To balance strict security with a seamless user experience, modern authentication relies on a two-token system: a short-lived **Access Token (the JWT)** and a long-lived **Refresh Token**.
            
            The lifespan of an access token (the JWT) must be kept incredibly short (typically 5 to 15 minutes) to minimizes the window of vulnerability if an attacker manages to intercept the token. However, forcing a user to manually enter their username and password every 15 minutes is an unacceptable user experience.
            
            The refresh token solves this dilemma. It is a long-lived, highly secure credential (often lasting days or weeks) whose exclusive purpose is to request new access tokens from the Auth service. When the short-lived access token naturally expires, the client application silently sends the refresh token to the Auth server behind the scenes. The server validates the refresh token, briefly checks the database to ensure the user’s account is still active and has not been suspended, and then issues a fresh access token. This maintains an uninterrupted session for the user while ensuring the server retains periodic control over authorization.
            
            Because refresh tokens are long-lived and capable of spawning new access tokens, they are high-value targets. If an attacker manages to exfiltrate a refresh token from a device, they could theoretically maintain infinite access to the system.
            
            To mitigate this catastrophic risk, the industry standard defense is **Refresh Token Rotation**. In this architecture, a refresh token is strictly single-use.
            
            When the client sends a valid refresh token to the Auth service, the server issues a new access token, but it *also* issues a brand new refresh token, instantly burning the old one. This single-use limitation acts as a cryptographic tripwire to detect theft.
            
            Consider this scenario: An attacker steals a refresh token and uses it to get a new access token. The server fulfills the request, but burns the original refresh token. Later, when the legitimate user's application naturally attempts to use that original, now-burned refresh token, the Auth server detects that a previously consumed token is being presented again. This is absolute proof that a token has been cloned or stolen.
            
            The Auth server's automated response is immediate and severe: it instantly revokes the entire "token family," destroying all active access and refresh tokens associated with that user session. Both the legitimate user and the attacker are immediately locked out and forced to re-authenticate from scratch using hard credentials, effectively neutralizing the breach.
            
            If your application requires a seamless "auto-renewal" experience where user activity infinitely delays expiration, you must abandon stateless tokens entirely. The only way to achieve sliding expiration is to return to stateful, session-based authentication, trading the scalability of statelessness for the convenience of server-managed state.
            

The output is three Base64-URL strings separated by dots that can be easily passed in HTML and HTTP environments, while being more compact when compared to XML-based standards such as SAML.

#### The Authentication Lifecycle

The lifecycle of a signed token requires explicit, stateless coordination between the client and the server. The process begins with issuance, where the user provides their credentials. Upon successful verification, the server aggregates the necessary user claims and generates the cryptographically signed token. [You also should not store sensitive info. All the information contained within the token is exposed to users or other parties, even though they are unable to change it. This means you should not put secret information within the token.] Once issued, the client receives the token and must decide how to store it—typically in application memory, local storage, or a browser cookie. For every subsequent request, the transmission phase occurs, where the client intentionally attaches the token to the outgoing network call. Finally, during validation, the receiving API intercepts the request, decodes the token, and verifies the cryptographic signature using its keys. If the signature is intact and the expiration claim is valid, the user is authenticated to proceed. 

Similarly to session Ids in session-based authentication, signed tokens like JWTs are completely transport-agnostic. The foundational specifications dictate the structure and cryptography of the payload, but they do not dictate how the token must be delivered. The client and server must agree on a method to transmit the token on every request, which typically boils down to the same two vectors we explored with session IDs.

If you choose to transmit the token via cookies, you are placing a stateless token inside a stateful browser mechanism. While dropping the token into an `HttpOnly` cookie effectively neutralizes Cross-Site Scripting (XSS) threats, it instantly reintroduces all the architectural friction we saw earlier. You are once again bound by strict browser rules, Cross-Origin Resource Sharing (CORS) headaches, and the absolute necessity of defending against Cross-Site Request Forgery (CSRF).

Alternatively, the token can be transmitted explicitly via HTTP response objects and headers. There is a widespread industry belief that tokens "must" or "should" be sent this way, but this reflex does not come from the token specification itself. It stems from RFC 6750—the OAuth 2.0 Authorization Framework specification for Bearer Token Usage. This is the document that standardized the `Authorization: Bearer <token>` HTTP header pattern. Because signed tokens are overwhelmingly utilized as bearer tokens in modern APIs, this header transmission method quickly became the undisputed default for distributed systems.

By sending tokens explicitly via headers rather than relying on automated browser cookies, developers bypass the browser's strict Same-Origin policies. CORS complexities are significantly reduced because the client application intentionally crafts and attaches the authorization header to cross-domain requests. This deliberate decoupling is what makes header-transmitted JWTs exceptionally well-suited for Single Sign-On (SSO) architectures, distributed microservices, and mobile applications where clients must seamlessly authenticate across multiple distinct domains.

#### Security and Architectural Trade-offs

While JWTs offer immense scalability, they introduce specific architectural challenges:

- **Public Payloads:** Because signed tokens are only Base64 encoded, anyone who intercepts the token can read its payload. You must **never** put secret information (like passwords or sensitive PII) inside a signed JWT.
- **Revocation Complexity:** Because the server does not check a database on every request, you cannot simply "delete" a session. Revoking a JWT before its natural expiration requires maintaining a server-side revocation list (blocklist) or waiting for the token to expire. For this reason, JWT lifespans should be kept short (e.g., 5–15 minutes), paired with a refresh token mechanism.
    
    For large-scale distributed systems that require instant revocation, the modern solution is Continuous Access Evaluation (CAE).
    
    Instead of forcing APIs to query a central database on every request, CAE flips the communication model using a publish/subscribe architecture. When a critical security event occurs—such as an administrator terminating an account—the Identity Provider immediately publishes a "Revocation Event." All relying APIs subscribe to this event feed and update a high-speed, local in-memory blocklist.
    
    When a request arrives, the API validates the token's cryptographic signature, checks the expiration, and then does a microsecond check against its local blocklist. If the token's unique ID (`jti` claim) or the user's ID (`sub` claim) is flagged, the API rejects the mathematically valid token. This allows the system to instantly revoke access across a massive fleet of microservices without ever executing a database query on the critical path.
    
- **Stale data in tokens:** While JWTs make *authentication* entirely stateless, they do not necessarily make your business logic stateless. Because token claims represent a snapshot of the user at the exact moment of issuance, they can quickly become stale. If your application requires strictly real-time authorization—such as checking if a user's account was suspended five seconds ago—you cannot rely solely on the token's payload and will still need to perform a database lookup.
- **Size Limits:** Including too many claims inflates the token size. If a token exceeds typical HTTP header limits (around 8 KB), web servers will reject the request.
- **Key Rotation:** Rotating your cryptographic signing keys will instantly invalidate all existing tokens signed by the old key, forcing all users to re-authenticate or refresh their tokens.

#### When Server-Side State is Still Necessary

Stateless tokens handle *identity*, but they are not a replacement for *state*. If your application requires multi-step workflows, shopping carts, or temporary UI state, that data should not be stuffed into a JWT. Instead, rely on the token strictly for identity and authorization, and use a dedicated datastore (like RAM, Redis or SQL) to handle application-specific session state.

#### Implementation in .NET

In ASP.NET Core, implementing JWTs involves transitioning away from cookie-based middleware. You will utilize `Microsoft.AspNetCore.Authentication.JwtBearer`. By registering the `AddJwtBearer` service, you configure the framework to automatically intercept incoming requests, validate the token's signature against your configured `TokenValidationParameters` (validating the Issuer, Audience, and Expiration), and reconstruct the user's `ClaimsPrincipal` for your protected routes.

### Encrypted + Signed Tokens

In the previous section, we established that while signed JSON Web Tokens (JWTs) provide strong guarantees of integrity and authenticity, they offer zero confidentiality. Because the payload is simply Base64Url-encoded, anyone who intercepts the token—or simply inspects it in their browser's developer tools—can read every claim inside it.

For many applications, this is perfectly fine. However, if your architectural requirements dictate that sensitive data, proprietary business logic, or internal system identifiers must be transported through the client without being exposed to the client, you cannot rely on a signed token alone. You need a token that guarantees integrity, authenticity, *and* confidentiality. This is achieved through a process of encrypting the data so it cannot be read, and then signing it so it cannot be tampered with.

#### JSON Web Encryption (JWE)

The industry standard for encrypted tokens is JSON Web Encryption (JWE - RFC 7516). While a JWS (Signed JWT) ensures the payload hasn't been changed, a JWE ensures the payload cannot even be seen by anyone other than the server that holds the decryption keys.

Instead of three Base64Url strings separated by dots, a JWE typically consists of five parts (Header, Encrypted Key, Initialization Vector, Ciphertext, and Authentication Tag). The mathematics and key management required to safely implement JWE are notoriously complex, involving multiple cryptographic algorithms working in tandem to encrypt the payload and secure the symmetric keys used for that encryption.

Because of this immense complexity, a deep dive into the JWE specification is out of scope for this guide. Instead, we will focus on how .NET developers practically solve the need for stateless, encrypted tokens using built-in framework capabilities.

#### The .NET Approach: The Data Protection Subsystem

If you are building exclusively within the ASP.NET Core ecosystem, you do not need to manually construct complex JWEs to achieve confidentiality. ASP.NET Core includes a powerful, built-in cryptographic engine called the **Data Protection subsystem**.

This subsystem provides a unified API for securing data. When used for authentication (such as with the default Cookie Authentication middleware, or when manually generating secure bearer tokens), the Data Protection API automatically handles the serialization, encryption, and signing of your user's identity.

**How does this work in the real world?**
The authentication lifecycle flows very similarly to the signed token approach, but the cryptographic mechanism differs:

1. **Issuance:** A client sends their credentials to your Auth service.
2. **Generation:** Your Auth service verifies the credentials and aggregates the user's claims. Instead of formatting this as a plain JSON string, it passes the identity to the Data Protection API.
3. **Encryption:** The subsystem automatically encrypts the payload using a symmetric encryption algorithm (like AES) so it is completely unreadable. 
4. **Signing**: It then cryptographically signs the encrypted blob (usually via HMAC) to ensure it cannot be tampered with.
5. **Transmission:** The result is a single, opaque, Base64-encoded blob called an identity ticket (often seen as the value inside an `.ASPXAUTH` cookie). This blob is returned to the client. The client stores it and attaches it to subsequent requests.
    
    Out of the box, ASP.NET Core tightly couples this encrypted identity ticket to its default Cookie Authentication handler. If you intend to use this encrypted blob as a stateless Bearer token for mobile apps or SPAs (transmitting it via HTTP headers), you will need to bypass the default cookie handler and implement custom middleware to extract and validate the token from the header.
    
6. **Validation:** When your API receives the request, it hands the opaque blob back to the Data Protection subsystem. The framework verifies the signature to ensure integrity. If the signature is valid, it decrypts the payload to ensure confidentiality, checks the expiration time, and finally rehydrates the user's `ClaimsPrincipal` into memory.

Because the encrypted blob already contains the full user identity, the server does not need to load session information from a database. Authentication remains fully stateless.

#### Security and Architectural Trade-offs

While the Data Protection subsystem provides a highly secure, developer-friendly path to stateless encrypted tokens, it introduces a vastly different set of architectural trade-offs compared to standard JWTs:

- **Complete Confidentiality:** (Pro) Unlike standard JWTs, the payload is completely opaque to the client. You can safely include internal identifiers or sensitive state data within the token without fear of exposure.
- **Automated Key Rotation:** (Pro) Unlike standard JWTs where you must manually manage key lifecycles, the Data Protection subsystem operates largely as a black box that automatically handles key generation, expiration, and rotation. It retains old keys just long enough to validate unexpired tokens, entirely removing this operational burden from the developer.
- **Proprietary Ecosystem Lock-in:** (Con) This is the most significant limitation. The .NET Data Protection format is proprietary to Microsoft. If your architecture includes microservices written in Node.js, Go, Python, or Java, they will not natively be able to decrypt or validate this token. It is strictly for communication between .NET applications.
- **Key Ring Management:** (Con) For standard JWTs, you only need to share a single HMAC secret (or a public/private key pair). The Data Protection subsystem requires a synchronized "Key Ring." If you are running multiple API servers behind a load balancer, they must all share the exact same cryptographic keys to decrypt the tokens. You must configure a centralized key store (like Redis, Azure Key Vault, or a shared network drive) to ensure all servers can read the opaque blobs.
- **Size Limitations:** (Con) Encryption adds cryptographic overhead (Initialization Vectors, Authentication Tags, etc.). Encrypted tokens are inherently larger than their plain-text JWT counterparts. If transmitted via HTTP headers, you must be extremely cautious not to exceed standard web server header limits (typically 8 KB).
- **Revocation Complexity:** (Con) Just like signed JWTs, these tokens are stateless. Revoking an encrypted token before its natural expiration still requires maintaining a server-side blocklist or relying on short lifespans and refresh tokens.

# Authorization

Once a user’s identity is authenticated, the system must determine if that user is actually allowed to perform the requested action. In modern software architecture, authorization begins with a single, foundational concept: **Claims**.

When an identity is created, it is assigned one or more claims issued by a trusted party (your Identity Provider or Auth service). A claim is simply a name-value pair that represents *what the subject is*, not *what the subject can do*.

Consider a real-world example: your driver's license. It was issued by a trusted local authority. Printed on that license is your date of birth. In software terms, the claim name is `DateOfBirth`, the claim value is `June 8th, 1990`, and the issuer is the DMV. If you want to gain access to a 21-and-over nightclub, the door security officer acts as the authorization middleware. They evaluate the value of your `DateOfBirth` claim, verify they trust the issuer, and grant or deny access based entirely on that data. They do not care about your name or your address; they only evaluate the specific claim required by their policy.

An identity can contain multiple claims with multiple values, and a system can issue multiple claims of the same type. How your application retrieves these claims depends entirely on your chosen authentication architecture:

- **Authorization with Basic Auth and Sessions:** Because these are stateful architectures, the incoming request only provides an identifier (credentials or a session ID). The server must take this identifier, query the Auth database to fetch the user’s associated claims, and hold them in memory.
- **Authorization with Tokens:** Because tokens are self-contained and stateless, all required identity information is packed into the token at the time of generation. When the request arrives, the server does not need to query a database. It simply validates the signature, decrypts the payload (if encrypted), and directly rehydrates the user's claims into memory.

Depending on your business requirements, you can evaluate claims using several different models:

- **Presence of a Claim (The Baseline):** The absolute simplest authorization model looks only for the *existence* of a claim, completely ignoring its value. For example, a system might grant access to a corporate intranet simply if the user possesses an `EmployeeBadgeNumber` claim. The actual number does not matter; holding the claim at all proves employment.
- **Claims as an Access Control List (ACL):** ACLs are an effective form of authorization for simple situations systems that do not require much granularity in controlling who can access what. It works by maintaining what is essentially a list of users and keeping track of whether or not they are allowed to perform a specific action. ACL asks: *"Is this user explicitly on the list?"* You can implement this statelessly by having your Auth service issue a claim that contains a literal list of accessible permissions or resources. For example, a token might contain the claim `"allowed_documents": ["doc1", "doc2", "doc3"]`. When the user attempts to access `doc2`, the application simply verifies that `doc2` exists within that claim's array.

    Note the labels **Users** and **Permissions**. A user can have zero or more permissions. With this design, permissions are relatively arbitrary, but so is the connection between the permissions and the users.
    
    ACL would be reasonably quick and cheap to implement and maintain in many systems. Systems with simple requirements and smaller scopes will benefit from adopting this design if they don’t need features offered by some of the more complex designs.
    
- **Claims as Role-Based Access Control (RBAC):** With RBAC, you can assign roles to your users and then make decisions for what they should be allowed to access based on these roles. RBAC asks: *"Does this user hold the correct title?"* Instead of querying a database for roles, your Auth service issues a claim named `role` with a value of `admin` or `finance`. Your application then checks for the presence of this specific claim value before allowing access to a restricted dashboard.
    
    The specifics of how roles work is down to the particular implementation. It’s possible to build things in a way where a single user can have multiple roles, or you can restrict users to a single role if that’s more appropriate for your use case.
    
    The following diagram is an example of what part of your Auth database schema might look like if you were to implement RBAC:
    
    ![relation-extended](https://cerbos.dev/blog-images/2021-11-09_3-most-common-authorization-designs-for-saa-s-products/saas-roles.png)
    
    Here, there are **Users**, **Roles**, and **Permissions**, and pivot tables between each of them. With this design, a user could be associated with zero or more roles, and each of those roles could subsequently be assigned zero or more permissions. These permissions would likely be fairly generic in scope, like `view_reporting` or `edit_users`, rather than being granular to the point of having permissions for individual resources.
    
    Because all permissions are essentially bound to roles, you don’t need to remember complex rules in order to grant new users access to things. Generally, you can just set a user up with the appropriate roles, and the rest should fall into place. This is in contrast to ACLs, where the rules themselves would likely need to be modified
    
    One of the biggest potential risks with RBAC is the possibility of a “role explosion.” This occurs when a need arises for increasingly granular control, and the only viable solution is to create more and more roles. If your system allows users to have multiple roles simultaneously, these new roles might only have a few permissions each. If not, it could result in a lot of duplicate roles with slight variations.
    
    Neither of these is an ideal scenario, and it highlights one of the big limitations of RBAC: it isn’t the best at doing granular permissions.
    
- **Claims as Attribute-Based Access Control (ABAC):** ABAC evaluates access based on specific user traits. Because claims *are* user attributes, this model can actually be incredibly simple. For instance, if your application restricts content to adults, it might just check if the `"age"` claim is 21 or greater.
    
    However, ABAC can also scale to handle massive complexity. ABAC uses attributes that belong to the objects involved in the transaction rather than discrete access records. If a token contains `"department": "finance"`, `"clearance_level": "secret"`, and `"region": "US"`, your application can evaluate all of those attributes simultaneously to determine if a user can view a highly classified document.
    
    Although it often takes more effort to implement well, it’s very powerful and often necessary for large organizations with complex business logic. The benefits of ABAC are abundant. By creating access policies based on assignable attributes, organizations have very granular and flexible control over who is allowed to access what. It also scales well, as policies do not need to be modified as new users come on board. Rather, the new users can simply be assigned the appropriate attributes. Access to the resources they need to do their job should follow.
    

**The Policy Pattern**

Whether you are checking for the mere presence of an employee badge, validating an array of ACL permissions, or calculating a complex ABAC scenario, your application needs a clean way to manage these rules. In modern software architecture, this is achieved through **Policies**.

A policy is a language-agnostic design pattern that separates authorization logic from routing and business logic. Instead of writing custom `if/else` statements inside every single API endpoint to manually inspect a user's claims, developers define declarative policies at application startup.

A policy acts as a named container that can combine *any* of the access models we just discussed. For example, you might create a policy named `RequireSeniorManagement`. The rules inside that policy might combine RBAC and ABAC, stating that the user must possess a `role` claim equal to `manager`, AND a `department` claim, AND a `years_of_service` claim greater than 5.

Once defined, you simply tag your protected resources or endpoints with the name of the policy. When a request arrives, the framework intercepts it, reads the user's claims, and evaluates them against the exact requirements of the requested policy. By cleanly decoupling the identity data (the claims) from the access rules (the policies), your application remains secure, scalable, and remarkably easy to audit.
